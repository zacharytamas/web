Lights-On Network

Because healthcare is too important to stay the same.

Lights On Network is a suite of dashboards and tools for Cerner Associates and Clients to monitor and manage the performance of Cerner's software and hardware solutions, as well as collecting data about the efficiency of their usage from an industry perspective down to an individual end-user.

Highlights

  • Played a key role in a JavaScript revolution, helping to improve the quality of existing and future code by championing JavaScript convention and build systems @TODO
  • Coordinating with a 20+ member team consisting of engineers, designers, strategists, and business analysts to effectively accomplish industry-leading corporate imperatives.

Skills Used/Developed on this Project

On this project we used Python with the Django framework backed by multiple Oracle databases. Our client-side styles were written in LESS and compiled to CSS and minified using django-pipeline.

Heavy amounts of JavaScript were used, with tests in the Jasmine testing framework, using frameworks such as Backbone.js, Underscore.js, jQuery, moment.js. Complex data visualizations were built using D3 with IE8 support through an internal project gone open-source called R2D3.

Case Study: Package Recommendations

One of my first projects after joining the team full-time at Cerner was to architect and implement the entire client-side portion of the overview page of a major project for Lights-On: the Package Recommendations dashboard.

The purpose of the Package Recommendations dashboard is to allow Cerner Associates and the I.T. departments of their Clients (commonly hospitals and doctors' offices) to manage their Cerner solutions. "Packages" recommended are typically patches that resolve bugs or increase performance, and the acceptance of these packages lead to more efficient healthcare for Cerner clients and their patients.

From the beginning there was a lot of hype about the project, as its planned functionality was an unprecedented amount of interactivity for a Lights-On dashboard. Even only having been on the team for two months, it was clear to me that the new JavaScript-centric direction many of our dashboards were headed that there needed to be an organized, focused driver of the effort. My prior experience with Backbone.js and proven skill with JavaScript led me to step forward and try my hand.

Starting with the approved design mockups from our design team, I took to outlining the design and breaking the highly interactive pieces into components and structuring them in such a way that could be built largely in isolation yet highly communicative using Backbone.js methodologies. From this architecting stage, a series of implementation plans was derived, which was broken into standalone "tickets" that could be done (mostly) independently.

During the following iteration cycle, I took on the client-side implementation of the plan meanwhile others on the team each built their respective pieces including an HTTP API for retrieving a user's recommendations and saving changes or notes to them. Through open communication, the team was able to pull together tightly and complete the project smoothly: a true shining moment for all of us on the front-end team.

As part of my vision for JavaScript applications on the team in the future, I took the opportunity to include Jasmine into the project and began writing the first JavaScript tests the codebase had seen. Following test-driven development as disciplined as I'd ever managed, the project was completed with virtually no defects and over a hundred Jasmine tests covering almost all JavaScript components and their interactions. This habit of writing Jasmine tests for all new JavaScript functionality has stuck, with everyone on the team contributing to a test suite that increases every development cycle.
