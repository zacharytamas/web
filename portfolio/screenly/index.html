Screenly

Digital signage made easy.

Highlights

  • Worked 100% remotely with team members across the globe, including California, London, Sweden, and Budapest.
  • Worked full stack from Python to browser-side coding to server administration (running Screenly on a Raspberry Pi).
  • Worked on both an open source version of the Screenly software stack, and a professional version for advertising agencies to manage a large amount of digital signs in their accounts.

The Product

While working for a startup named Wireload, I contributed to the development of two versions of Screenly: an open source version and a professional, commercial version of the product with premium features.

Both versions of Screenly are centered around specialized software which runs on Raspberry Pi microcomputers to present web pages, image slideshows, and 1080p video to any display that can be driven via HDMI 2.0 or RCA composite cable.

The free, open source version of Screenly allows for a technically inclined user to easily setup their own Raspberry Pi, connect it to the Internet, and manage the content being shown on it by logging directly into it via a web interface.

Screenly Pro consists of a centralized, multi-tenant server cluster which periodically syncs with all the Screenly OSE screens across the world that it is configured to manage. Customers pay Screenly to allow them to manage all of their Screenly-powered screens from one interface instead of logging into individual screens to update content. Screenly Pro allows you to add content and spread it across all of your screens in almost realtime, converting videos and optimizing imagery before automatically pushing the latest content directly where it needs to go.

On this project we used Python with the Bottle framework backed by MongoDB. Our client-side styles were written in LESS and compiled to CSS.

While on this project I learned CoffeeScript and Backbone.js. As one of my first major JavaScript-heavy projects, I rapidly self-educated myself with all JavaScript resources that I could find. While most of my time was spent with the front-end scripting, I had my hands in some of the Python and video playing components of the project, deploying code directly to a Raspberry Pi.

Case Study: Dealing With Limited Resources

Engineering software for Screenly was an interesting new challenge because you became very well aware of the limitations of the hardware. Generally those of us engineers who work on the web aren't overly concerned with the performance of every line of code as servers running the code or users' machines running your JavaScript are reasonably well-resourced in general.

This was not the case at all for Screenly. Because the core unit of the Screenly system is a server that runs on a Raspberry Pi, efficient algorithms and processes were absolutely paramount in its development. An operation which on a typical desktop computer might take seconds may perhaps take a minute or more on the limited ARM processor in a $35 Raspberry Pi. Special care was taken in writing efficient syncing algorithms to manage the Pis remotely without bogging them down, especially when they could at any moment be rendering and displaying 1080p HD video.

Another less obvious resource concern was the storage of a Pi: there isn't any. Raspberry Pis' primary source of storage are SD cards such as you might use in a digital camera. Due to the regular syncing of Screenly and downloading/updating/deleting of media, care was taken to be good stewards of storage to protect the health of the SD cards as much as possible so that their lifespans were maximized.
